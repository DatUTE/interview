# CI/CD Overview

CI/CD is the automation flow that builds, tests, packages, and sometimes deploys software whenever code changes.

In embedded projects, CI/CD often produces binaries, firmware, SDK packages, or full Linux images instead of deploying directly to cloud servers.

## Core Ideas

### Continuous Integration

Continuous Integration means every change is merged frequently and validated automatically.

Typical checks:

- Compile/build
- Unit tests
- Static analysis
- Formatting or linting
- Dependency checks
- Packaging artifacts

### Continuous Delivery

Continuous Delivery means the software is always in a releasable state. The pipeline can produce release artifacts, but deployment to target hardware may still require approval.

### Continuous Deployment

Continuous Deployment means every approved change can be deployed automatically. In embedded systems, this is less common than backend systems because deployment may involve real hardware, safety checks, flashing, or lab infrastructure.

## Typical Embedded Pipeline

```text
checkout source
  -> setup toolchain/container
  -> build C/C++ code
  -> run unit tests
  -> run static analysis
  -> package binary artifacts
  -> build Yocto image or firmware bundle
  -> deploy to test target
  -> run integration/SIL/HIL tests
```

## Common Pipeline Stages

| Stage | Purpose |
|---|---|
| build | Verify the code compiles |
| test | Run unit or component tests |
| static-analysis | Catch style, bug, security, or MISRA-like issues |
| package | Produce deployable artifacts |
| image-build | Build firmware or Linux image |
| deploy | Flash or copy artifacts to target |
| integration-test | Validate on simulator or hardware |

## Artifacts

Artifacts are files generated by the pipeline and passed between stages.

Examples:

- Executable binaries
- Shared libraries
- Unit test reports
- Coverage reports
- Yocto images
- Root filesystem archives
- Flashable firmware packages

## CI Gates

A gate is a required check before merging or releasing.

Common gates:

- Build must pass
- Unit tests must pass
- No critical static analysis issue
- Code review approval
- Integration test pass
- Release approval

## Pipeline Design Principles

A good pipeline should be:

- Fast enough to give developers useful feedback.
- Reliable enough that failures are trusted.
- Reproducible across local and CI environments.
- Split into clear stages with readable logs.
- Able to publish artifacts for later debugging.
- Strict for release branches and flexible for early development branches.

For embedded projects, avoid putting every slow hardware test on every commit. Use layered validation:

```text
every commit: build + unit tests + static checks
merge request: integration build + selected tests
nightly: full image build + longer tests
release candidate: HIL + flashing + system validation
```

## Example GitLab CI Pipeline

This example shows a common embedded C/C++ pipeline shape.

```yaml
stages:
  - build
  - test
  - package
  - image

variables:
  BUILD_DIR: build

build_app:
  stage: build
  image: embedded-build:dev
  script:
    - cmake -S . -B "$BUILD_DIR"
    - cmake --build "$BUILD_DIR" --parallel
  artifacts:
    paths:
      - build/
    expire_in: 1 week

unit_tests:
  stage: test
  image: embedded-build:dev
  script:
    - ctest --test-dir "$BUILD_DIR" --output-on-failure
  needs:
    - build_app

package_app:
  stage: package
  image: embedded-build:dev
  script:
    - cmake --install "$BUILD_DIR" --prefix package-root
    - tar -czf app-package.tar.gz package-root
  artifacts:
    paths:
      - app-package.tar.gz

yocto_image:
  stage: image
  image: yocto-build:dev
  script:
    - source poky/oe-init-build-env build
    - bitbake company-image
  artifacts:
    paths:
      - build/tmp/deploy/images/
```

Key ideas:

- `stages` define ordering.
- `image` controls the build environment.
- `artifacts` preserve output for later stages or download.
- `needs` lets jobs depend on earlier jobs more explicitly.

## Artifacts vs Cache

Artifacts and cache are often confused.

Artifacts are outputs of a pipeline that should be preserved or passed between jobs.

Examples:

- Test reports
- Binaries
- Packages
- Yocto images
- Coverage reports

Cache is reused data that speeds up future jobs.

Examples:

- Compiler cache
- Downloaded dependencies
- Yocto `downloads`
- Yocto `sstate-cache`

Simple rule:

```text
artifact = output you care about
cache = temporary speed-up data
```

## Branch Strategy

Common branch types:

- Feature branch: developer work.
- Main branch: integrated development.
- Release branch: stabilized version.
- Hotfix branch: urgent fix for released version.

Typical policy:

- Feature branches open merge requests.
- Main branch must stay buildable.
- Release branches have stricter gates.
- Tags identify release candidates and official releases.

For automotive or regulated environments, the release branch and tag should be traceable to test results and produced binaries.

## Test Strategy

Different tests should run at different pipeline levels.

Fast checks:

- Formatting
- Linting
- Unit tests
- Compile checks

Medium checks:

- Component tests
- Integration tests
- Cross-compiled target build
- Static analysis

Slow or expensive checks:

- Full Yocto image build
- Flashing target hardware
- HIL tests
- Long-running stress tests

Good pipeline design keeps fast checks early so obvious failures stop quickly.

## Static Analysis And Quality Checks

Embedded CI often includes:

- Compiler warnings with `-Wall -Wextra -Werror` for selected modules
- `clang-tidy`
- `cppcheck`
- MISRA-like checks
- Code formatting
- Dependency/license scanning
- Unit test coverage

Do not blindly make every warning fatal on legacy code. A practical approach is to enforce strict checks on new or touched code first.

## Deployment And Flashing Stage

Deployment in embedded CI may include:

```text
reserve hardware target
  -> power cycle board
  -> flash image
  -> wait for boot
  -> collect serial logs
  -> run smoke test
  -> publish logs and result
```

Important concerns:

- Only one job should control a hardware target at a time.
- The pipeline should collect logs even when flashing or boot fails.
- Hardware jobs often need retry logic, but retries should not hide real bugs.
- Lab runners should be protected if they can access expensive or sensitive hardware.

## Release Pipeline

A release pipeline usually adds more control than a normal merge request pipeline.

Common release steps:

```text
build from clean source
  -> run full validation
  -> produce artifacts
  -> create version metadata
  -> archive logs and reports
  -> tag release
  -> publish release package
```

Release metadata should include:

- Git commit SHA
- Git tag
- Pipeline ID
- Build time
- Toolchain version
- Docker image version
- Yocto layer revisions
- Machine/distro configuration

## Common CI/CD Failure Modes

- Pipeline passes locally but fails in CI because environments differ.
- Cache hides missing dependency problems.
- Artifact paths are wrong.
- Test job runs before build artifact is available.
- Hardware target is offline or already used by another job.
- Yocto build fails because `sstate-cache` or `downloads` path is misconfigured.
- Release artifact cannot be traced to exact source and config.

## Embedded-Specific Challenges

- Cross-compilation is slower and more complex than native builds.
- Hardware test labs can be limited or shared.
- Builds may require exact toolchain versions.
- Debugging failed flashing or boot can require serial logs.
- Pipeline output must be traceable to source commit and configuration.

## Interview Notes

Good answers should mention:

- CI gives fast feedback before code is merged.
- Docker helps make build environments reproducible.
- Artifacts should be versioned and traceable.
- Embedded CD often stops at "ready to deploy" because real hardware deployment may need manual or lab-controlled steps.
- HIL tests are expensive, so they may run nightly or on release candidates instead of every commit.
